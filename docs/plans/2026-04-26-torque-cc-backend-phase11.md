> **For Claude:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` to implement this plan task-by-task. Phase 11 lifts the structural emitter prerequisite that Phase 10's Task 10.0 v2 triage discovered (Phase 9B's struct-vs-tuple inconsistency between helper signatures, body access, and call-site emission). Once that's reconciled, the NewPromiseCapability bridge + a small (a)-bucket shim cluster + (c)-bucket NamespaceConstants close out PromiseTry.

**Goal:** Ship `PromiseTry` (`promise-try.tq`) as the first real-world combined catch_block + varargs forcing target. Phase 10 closed with Task 10.0.1 (StoreReference non-smi-tagged emission) shipped + the broader scope deferred. Phase 11 picks up the deferred work.

**Architecture (target):**

The Phase 9B varargs lowering chose `std::tuple<...>` as the C++ representation for torque struct types in the kCCBuiltins emission, mirroring how `Type::GetRuntimeType()` lowers structs in the kCC/kCCDebug paths. But the helper-body access pattern (generated by `LowerParameter` recursive descent in implementation-visitor.cc:4072-4088) uses `<param>.<field>` named-field syntax, which is invalid on tuples. The CSAGenerator path doesn't hit this because it has `TorqueStructArguments` and friends as real C++ structs in `csa-types.h` (auto-emitted from torque struct declarations).

Phase 11 reconciles this by giving kCCBuiltins emission a parallel struct-definition pass — same architectural pattern as `csa-types.h`, but emitted into the per-TU `*-tq-ccbuiltins.cc` preamble with C++-native field types (`void*`, `Address`, `intptr_t`, `Tagged<T>`) instead of `TNode<RawPtrT>` etc. After Phase 11:

- Helper signatures use struct names (e.g., `TorqueStructArguments p_arguments`) instead of `std::tuple<...>`.
- Helper bodies' `<param>.<field>` access compiles cleanly against the named struct.
- Call sites emit `TorqueStructArguments{a, b, c, d}` aggregate-init instead of `std::make_tuple(...)`.
- Phase 9B's prelude rewires correctly: `TorqueStructArguments torque_arguments{nullptr, address_of_first_argument(), length, actual_count};`.

This matches the spirit of V8's existing Torque CSA backend — a parallel struct emission pass keyed on `OutputType::kCCBuiltins`. Future V8 upgrades that touch struct types in csa-types.h will surface as visible diffs against our parallel kCCBuiltins struct-emission pass.

**Tech Stack:** Same as Phase 10 — V8 Torque (C++), Torque-generated C++, d8, gtest (cctest), bash. No wasm-posix-kernel changes.

## Task 11.1: Phase 9B struct/tuple emitter reconciliation

The crux: the kCCBuiltins emission emits struct types as `std::tuple<flat_field_types...>` everywhere a runtime type is needed (helper signatures, return types, stack temporaries), AND emits struct-field access as `<param>.<field_name>` syntax (LowerParameter recursive naming). These two are inconsistent — `<tuple_value>.<field_name>` doesn't compile.

**Fix architecture (mirrors CSAGenerator's csa-types.h pattern):**

### 11.1a: Add an OutputType-aware struct-runtime-type method

`Type::GetRuntimeType(OutputType)` overload (or a new method like `GetCCBuiltinsRuntimeType()`). For struct types in `OutputType::kCCBuiltins`, return `(*struct_type)->GetGeneratedTypeName()` (e.g., `"TorqueStructArguments"`). For other modes, preserve current `std::tuple<...>` lowering. Located in `types.cc:1444-1462`.

The choice of overload-vs-new-method depends on call-site audit:

```bash
grep -nE "GetRuntimeType\b" deps/v8/src/torque/cc-generator.cc deps/v8/src/torque/implementation-visitor.cc | wc -l
```

(~30 callsites in Phase 10 baseline). Each callsite needs review for whether it's emitting kCCBuiltins output or kCC/kCCDebug output. The cleanest path: a new method `Type::GetRuntimeTypeForCCBuiltins()` invoked only from the kCCBuiltins-emission paths in `GenerateFunction` (lines 2479, 2504), `Visit(Builtin*)` for the JS-linkage builtin path (line 850, 855, 884, 914, 916), and the CCGenerator body emission paths under `is_cc_builtins_` (cc-generator.cc:99, 120, 141, etc.).

Estimated surface: ~40 LoC of edits across ~20 callsites + new method body (~8 LoC).

### 11.1b: Auto-emit struct definitions per-TU

In implementation-visitor.cc's per-TU emission (around line 2225 where `cc_builtins_preamble` is assembled), walk `TypeOracle::GetAggregateTypes()`, filter for `StructType`, and emit each as a C++ struct.

```cpp
// Phase 11 — per-TU mirror of CSA's csa-types.h struct emissions, but with
// C++-native field types instead of TNode<>. Each torque struct is emitted
// once per TU under an include guard. The struct name matches CSAGenerator's
// (StructType::GetGeneratedTypeName, e.g., "TorqueStructArguments") so future
// emitter changes can swap representations transparently.
namespace v8::internal {

#ifndef V8_INTERNAL_DEFINED_TorqueStructArguments
#define V8_INTERNAL_DEFINED_TorqueStructArguments
struct TorqueStructArguments {
  void* frame;
  Address base;
  intptr_t length;
  intptr_t actual_count;
};
#endif

#ifndef V8_INTERNAL_DEFINED_TorqueStructArgumentsIterator_0
#define V8_INTERNAL_DEFINED_TorqueStructArgumentsIterator_0
struct TorqueStructArgumentsIterator_0 {
  TorqueStructArguments arguments;
  intptr_t current;
};
#endif

// ... etc, one struct per StructType ...

}  // namespace v8::internal
```

For each struct, recursively use the new `GetRuntimeTypeForCCBuiltins()` for field types. Nested struct fields work because the inner struct is also emitted (in dependency order — TopoSort the StructTypes by reference if needed).

**Scope concern**: V8 has ~349 struct types (per Phase 10 triage of csa-types.h). Emitting all in every kCCBuiltins TU bloats output significantly. Two options:

- **(I) Emit unconditionally**: simplest, mirrors csa-types.h. Compile cost is small (struct defs are zero-cost at runtime). Diff impact: every kCCBuiltins .cc gets +~3-4 KB of struct decls.
- **(II) Emit only struct types reachable from this TU's kCCBuiltins emissions**: collect during emission, emit in preamble. More precise, more code.

Recommendation: start with **(I)** for Phase 11. If the patch-line-count delta is excessive, iterate to (II) in a follow-on. The 18 fixture goldens will all gain ~349 struct decls — refresh once, audit second-run byte-exactness.

Estimated surface: ~30 LoC of emitter loop + recursive type rendering.

## Task 11.3: Macro shims (Call variadic, GetArgumentValue, ThrowTypeError MT, IntPtrSub, WordEqual)

Add to `runtime-macro-shims.h` (the (a)-bucket slot for trivial macro shims):

### `Call` — variadic JS-linkage call

Wrap `Execution::Call` (`deps/v8/src/execution/execution.h:26`). Variadic template to handle 3-arg, 4-arg, 6-arg call sites that PromiseTry emits.

```cpp
template <typename... Args>
inline Tagged<JSAny> Call(Tagged<Context> context, Tagged<JSAny> callable,
                          Tagged<JSAny> receiver, Args... args) {
  USE(context);
  Isolate* isolate = Isolate::Current();
  Handle<Object> callable_h(callable, isolate);
  Handle<Object> receiver_h(receiver, isolate);
  std::array<DirectHandle<Object>, sizeof...(args)> arg_handles{
      DirectHandle<Object>{args, isolate}...};
  base::Vector<const DirectHandle<Object>> args_vec(
      arg_handles.data(), arg_handles.size());
  DirectHandle<Object> result;
  if (!Execution::Call(isolate, callable_h, receiver_h, args_vec)
           .ToHandle(&result)) {
    // Exception path — caller's catch_block handles via existing
    // has_exception() check.
    return UncheckedCast<JSAny>(
        ReadOnlyRoots(isolate).undefined_value());
  }
  return Cast<JSAny>(*result);
}
```

Estimated surface: ~20 LoC (one variadic template covers all arities).

## Task 11.4: Builtin_NewPromiseCapability bridge (fast-path-only)

Per Phase 10's "Decision points" discussion (verification.md Phase 10 Summary), this is the budget exception: a TFC bridge that justifies >50 LoC because (a) V8 has no public `Factory::NewPromiseCapability` C++ helper, (b) the bridge is reusable across the entire Promise family (Promise.try, Promise.all, Promise.any, Promise.race, Promise.allSettled, Promise.withResolvers).

**Implementation: fast path only** — assume `constructor == native_context.promise_function()`. UNREACHABLE on subclass-Promise slow path.

```cpp
// Phase 11 — Builtin_NewPromiseCapability TFC bridge.
//
// V8's `NewPromiseCapability` is a TFC builtin (CSA-emitted machine code).
// No `Builtin_NewPromiseCapability` C symbol exists in any libv8.* lib
// (TFC is "Torque, Function, Custom-call-descriptor" — non-JS-linkage,
// not Execution::Call-invocable from C++). The closest C++-callable APIs
// are isolate->factory()->NewJSPromise() + the closure-context machinery
// in promise-abstract-operations.tq:345-376.
//
// We replicate the FAST path of InnerNewPromiseCapability — the case
// where `constructor == native_context.promise_function()`. Real-world
// callers (Promise.try, Promise.all, Promise.any, ...) always pass
// the global Promise as `this`, so the slow path (custom Promise
// subclass — `class MyPromise extends Promise`) is unreachable in
// practice for the targets Phase 11+ whitelists. UNREACHABLE on the
// slow path with a `// TODO(phase-12+)` marker.
//
// Closure-context allocation: `CreatePromiseResolvingFunctions` in
// promise-abstract-operations.tq:325 allocates a 3-slot
// PromiseResolvingFunctionContext + 2 closures (resolve, reject)
// referencing it. We use V8's existing allocators:
//   - Factory::NewFunctionContext(...) for the closure context.
//   - Factory::NewFunctionFromSharedFunctionInfo for the closures

//     PromiseCapabilityDefaultRejectSharedFun (root SFIs).
//

  // Fast path: replicate InnerNewPromiseCapability's
  // `constructor == promise_function` branch.
  bool dbg = IsTrue(debug_event, isolate);
  Handle<JSPromise> promise = isolate->factory()->NewJSPromise();

  // Exact sequence TBD from a spike — see Phase 11 close criteria below.
  // Conservative LoC estimate: 50-80 LoC including helper-allocations.
```
