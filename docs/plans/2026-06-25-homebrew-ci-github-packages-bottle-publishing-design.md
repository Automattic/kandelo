# Homebrew CI And GitHub Packages Bottle Publishing

Date: 2026-06-25

Tracked work:

- `kd-7dk` - Design Homebrew CI + GitHub Packages bottle publishing path.
- `kd-8ho` - Homebrew CI and GitHub Packages publishing path.

## Position

Kandelo should publish Homebrew bottles from a normal Homebrew tap, with
trusted CI producing the bottle bytes, formula bottle metadata, Kandelo sidecar
metadata, and validation evidence together.

Use GitHub Packages as the canonical bottle-byte host for guest `brew install`
because that is Homebrew's supported path for tap bottles generated by
`brew tap-new --github-packages` and for Homebrew-maintained bottles. Use
GitHub Releases in the same tap repository for Kandelo-specific sidecars,
metadata indexes, provenance reports, and optional precomposed VFS images. Do
not use the Kandelo package-source `index.toml` as a substitute for Homebrew's
`bottle do` block; the two indexes serve different consumers.

The first milestone may temporarily publish the `hello` bottle to GitHub
Releases with a formula `root_url` only if the GitHub Packages upload path is
blocked by the Kandelo bottle tag/platform patch. That fallback must be an
explicit owner decision and should keep the same sidecar schema so migrating to
GitHub Packages is URL-only.

## Evidence And Gap

This bead is design work rather than a runtime defect. The reproducible gap in
the current tree is:

- `docs/package-management.md` and `docs/binary-releases.md` define Kandelo's
  package archive release flow: `.tar.zst` package archives, release
  `index.toml`, `cache_key_sha`, and `scripts/index-update.sh`.
- `docs/package-sources.md` defines external Kandelo package sources and
  reusable package-source publishing, but that publishes Kandelo package
  archives, not Homebrew bottles or formula `bottle do` metadata.
- The existing Homebrew plan defines the runtime and VFS-builder direction, but
  it does not specify the trusted tap CI path that builds bottles, publishes to
  GitHub Packages, generates sidecars, validates Node/browser behavior, and
  records rollback state.

The design below fills that gap without changing runtime behavior.

## Non-Goals

- Do not implement a Kandelo-specific `brew` clone.
- Do not evaluate Formula Ruby in host VFS builders, browser galleries, shared
  URL paths, or untrusted metadata readers.
- Do not make GitHub Packages the storage format for Kandelo sidecar metadata
  unless a later implementation proves a real need. Releases are simpler for
  generic JSON, reports, and VFS images.
- Do not replace Kandelo's existing package archive release flow. Homebrew
  bottles are an additional distribution unit.
- Do not claim full Homebrew support from a successful `hello` bottle.
- Do not hide ABI drift with compatibility shims. Rebuild stale bottles.

## Repository Shape

Create a dedicated tap repository, tentatively:

```text
Automattic/homebrew-kandelo
```

The tap should stay recognizable to Homebrew maintainers:

```text
README.md
Formula/
  hello.rb
  zlib.rb
Kandelo/
  metadata.schema.json
  link-manifest.schema.json
  metadata.json
  formula/
    hello.json
  link/
    hello-2.12.1-rebuild0-wasm32.json
  reports/
    hello-2.12.1-rebuild0-wasm32.provenance.json
.github/workflows/
  tests.yml
  publish-bottles.yml
  rebuild-bottles.yml
  validate-metadata.yml
```

`Formula/` is the user-facing Homebrew contract. Formulae should use standard
Homebrew DSL wherever possible: `depends_on`, `resource`, `patch`, `bottle do`,
`revision`, `rebuild`, and `test do`.

`Kandelo/` is generated and reviewed metadata for tools outside the guest:

- `metadata.json` is the top-level Kandelo/Homebrew index for the tap commit.
- `formula/<name>.json` carries one formula's Kandelo bottle metadata.
- `link/<formula>-<version>-rebuild<N>-<arch>.json` tells the host VFS builder
  how to pour and link a bottle without evaluating Formula Ruby.
- `reports/` carries provenance and validation evidence from the publishing
  workflow.

The publish workflow should upload the same generated `Kandelo/` payload to a
tap GitHub Release named:

```text
bottles-abi-v<ABI_VERSION>
```

The checked-in files make metadata reviewable. The release assets make a stable
fetch target for VFS builders and automation that should not read a moving
branch.

## Artifact Storage Model

| Artifact | Canonical storage | Why |
|---|---|---|
| Formula source | Tap Git repository | Normal Homebrew review and tap trust model. |
| Bottle tarballs | GitHub Packages through Homebrew's bottle upload path | Guest `brew install` should consume Homebrew-native bottle URLs and formula bottle blocks. |
| Kandelo metadata index | Tap GitHub Release `bottles-abi-v<N>` and checked-in `Kandelo/metadata.json` | Generic JSON should be easy for host builders, CI, and humans to fetch and review. |
| Link/install manifests | Same as Kandelo metadata | Host VFS builders need structured metadata without Ruby execution. |
| Provenance reports | Same as Kandelo metadata | Publication evidence must survive workflow log expiry. |
| Precomposed VFS images | GitHub Release assets, not GitHub Packages | VFS images are generic launch artifacts and map to Kandelo's existing browser-gallery model. |
| PR-only bottle outputs | Workflow artifacts | Untrusted PRs should not write packages or releases. |

GitHub Packages URLs should be treated as bottle-byte implementation detail in
the formula `bottle do` block and sidecar metadata. The VFS builder should
consume the sidecar's URL and sha256 rather than guessing GitHub Packages paths.

## Metadata Contract

The top-level metadata is single ABI:

```json
{
  "schema": 1,
  "tap": "kandelo/core",
  "tap_repository": "Automattic/homebrew-kandelo",
  "tap_commit": "<git-sha>",
  "kandelo_abi": 15,
  "generated_at": "2026-06-25T00:00:00Z",
  "generator": "kandelo-homebrew-publish 1",
  "packages": [
    {
      "name": "hello",
      "full_name": "kandelo/core/hello",
      "version": "2.12.1",
      "formula_revision": 0,
      "bottle_rebuild": 0,
      "formula_path": "Formula/hello.rb",
      "dependencies": [],
      "bottles": [
        {
          "arch": "wasm32",
          "bottle_tag": "wasm32_kandelo",
          "kandelo_abi": 15,
          "cellar": "/home/linuxbrew/.linuxbrew/Cellar",
          "prefix": "/home/linuxbrew/.linuxbrew",
          "url": "https://...",
          "sha256": "<64-hex>",
          "bytes": 12345,
          "cache_key_sha": "<64-hex>",
          "link_manifest": "Kandelo/link/hello-2.12.1-rebuild0-wasm32.json",
          "runtime_support": ["node", "browser"],
          "browser_compatible": true,
          "fork_instrumentation": "not-required",
          "status": "success",
          "built_by": "https://github.com/.../actions/runs/<id>",
          "built_from": {
            "kandelo_repository": "Automattic/kandelo",
            "kandelo_commit": "<git-sha>",
            "tap_commit": "<git-sha>",
            "formula_sha256": "<64-hex>"
          }
        }
      ]
    }
  ]
}
```

Required bottle fields:

- `arch`: `wasm32` or `wasm64`.
- `bottle_tag`: the Homebrew platform tag used in the formula `bottle do`
  block. Start with `wasm32_kandelo` and `wasm64_kandelo`; add an ABI-bearing
  tag only if the patched Homebrew tag parser can round-trip it cleanly.
- `kandelo_abi`: the guest ABI epoch. It must match the release tag and the
  in-tree `ABI_VERSION` used to build the bottle.
- `url` and `sha256`: the bottle byte address and integrity check.
- `cellar` and `prefix`: the supported Homebrew layout. The default is
  `/home/linuxbrew/.linuxbrew`.
- `cache_key_sha`: the Kandelo package/build input equivalence hash for the
  Wasm artifact included in the bottle.
- `link_manifest`: the structured install/link description used by host VFS
  builders.
- `runtime_support` and `browser_compatible`: explicit Node/browser support.

Optional status fields should mirror Kandelo's binary ledger semantics:

- `status = "success"` means the current bottle fields are authoritative.
- `status = "failed"` records the latest failed rebuild and keeps `fallback_*`
  fields for the last successful bottle.
- `status = "pending"` or `"building"` means a rebuild is queued or running;
  consumers may use a last-green fallback if present.

## Homebrew And Kandelo Version Mapping

Do not conflate Homebrew identity with Kandelo build identity:

- Homebrew `version`, formula `revision`, and bottle `rebuild` control what
  `brew install` considers current for a formula.
- Kandelo `ABI_VERSION` controls guest/kernel binary compatibility.
- Kandelo `cache_key_sha` controls whether the published Wasm artifact is built
  from the exact recipe, SDK, sysroot, toolchain, and transitive dependency
  inputs this checkout expects.
- Kandelo `build.toml` `revision` remains package-cache invalidation. When a
  package output changes and a Homebrew formula bottles that output, either the
  formula revision or bottle rebuild must move so Homebrew fetches new bytes.

ABI bump policy:

1. Publish to a new `bottles-abi-v<N+1>` release.
2. Generate metadata with `kandelo_abi = N+1`.
3. Rebuild every bottle whose `cache_key_sha` includes the ABI.
4. Keep the old `bottles-abi-v<N>` release as historical state.
5. Reject mixed-ABI metadata before publish and before VFS image construction.

## Link Manifest Contract

The link manifest is required for published VFS images. The host VFS builder
must not infer Homebrew's link behavior from formula Ruby.

Example:

```json
{
  "schema": 1,
  "package": "hello",
  "version": "2.12.1",
  "arch": "wasm32",
  "kandelo_abi": 15,
  "prefix": "/home/linuxbrew/.linuxbrew",
  "cellar": "/home/linuxbrew/.linuxbrew/Cellar",
  "keg": "/home/linuxbrew/.linuxbrew/Cellar/hello/2.12.1",
  "links": [
    {
      "type": "symlink",
      "source": "Cellar/hello/2.12.1/bin/hello",
      "target": "bin/hello"
    }
  ],
  "receipts": [
    "Cellar/hello/2.12.1/.brew/hello.rb",
    "Cellar/hello/2.12.1/INSTALL_RECEIPT.json"
  ],
  "env": {
    "PATH_prepend": ["bin", "sbin"]
  }
}
```

Validation rules:

- Reject absolute `source` or `target` paths inside link entries.
- Reject `..`, empty path segments, duplicate targets, and links escaping the
  prefix.
- Reject links to files missing from the bottle payload.
- Reject unsupported node types. Start with regular files, directories, and
  symlinks.
- Require receipts for published images. Fixture-only bypasses must be explicit
  debug flags.

## Trusted CI Flow

Use three workflow classes.

### 1. Pull Request Validation

Trigger:

```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened]
permissions:
  contents: read
```

Responsibilities:

1. Checkout the tap PR and the selected Kandelo ref.
2. Enter Kandelo's `scripts/dev-shell.sh`.
3. Build the Kandelo sysroot and SDK pieces needed by the changed formulae.
4. Run Homebrew audit/style checks for the tap.
5. Build changed bottles as workflow artifacts only.
6. Generate sidecar metadata and link manifests in a temporary directory.
7. Validate metadata schemas, ABI, arch, paths, URL shape, sha256, and
   dependency closure.
8. Run formula tests through a Kandelo Node harness. Do not execute Wasm
   binaries as host Linux binaries.
9. For browser-compatible packages, build or reuse a precomposed VFS image and
   run the browser smoke path.

Pull requests from forks must not receive `packages: write` or `contents:
write`. If fork CI cannot build because secrets or package write permissions
are unavailable, it should still run schema/audit checks and clearly report the
skipped publish-only validation.

### 2. Trusted Publish

Trigger:

```yaml
on:
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      packages:
        description: Comma-separated package names, or all.
      kandelo-ref:
        description: Kandelo commit/ref to build against.
permissions:
  contents: write
  packages: write
  id-token: write
```

Responsibilities:

1. Compute the publish matrix from changed formulae, dependency closure, target
   arch list, and current Kandelo ABI.
2. Reuse Kandelo's dev shell, SDK activation, sysroot build, and
   `cache_key_sha` logic.
3. Build one `(formula, arch)` per matrix entry.
4. Run `brew install --build-bottle` with the Kandelo SDK/toolchain available.
5. Run the formula's `test do` through Kandelo, not by host execution.
6. Run `brew bottle --json` and generate the formula `bottle do` block.
7. Upload bottle bytes with Homebrew's GitHub Packages path.
8. Generate sidecar metadata, link manifests, and provenance reports.
9. Commit generated bottle blocks and `Kandelo/` metadata back to the tap, or
   open a bot PR if direct pushes to `main` are disallowed.
10. Upload the generated `Kandelo/` payload to `bottles-abi-v<N>`.
11. Mark failures in metadata without deleting the last-green bottle.

Use a concurrency group scoped to the tap and target ABI:

```text
homebrew-kandelo-publish-bottles-abi-v<N>
```

That serializes metadata writes without blocking unrelated Kandelo package
source publishes.

### 3. Manual Rebuild And Repair

Trigger:

```yaml
on:
  workflow_dispatch:
    inputs:
      packages:
      kandelo-ref:
      force:
      repair-only:
```

Responsibilities:

- Rebuild selected formulae when a cache-key input was wrong or when an ABI bump
  needs a fresh durable release.
- Regenerate metadata from existing bottle bytes when only metadata validation
  or release repair is needed.
- Preserve last-green fallback fields when a rebuild fails.
- Refuse to publish if the formula bottle block, sidecar metadata, release ABI,
  and bottle archive sha disagree.

## Provenance And Trust

Trust boundaries:

- Formula Ruby executes only in trusted tap CI or inside the user's Kandelo VM.
- Host VFS builders consume JSON metadata and verified tarballs only.
- Browser/gallery code consumes bounded descriptors, published VFS images, and
  metadata that has already passed host validation.

Each published bottle should record:

- tap repository and commit;
- Kandelo repository and commit;
- GitHub Actions run URL and job ID;
- Homebrew/brew version or commit;
- Kandelo ABI;
- target arch;
- formula path and sha256;
- package cache key sha;
- bottle sha256 and byte size;
- build timestamp derived from the source commit, not wall-clock time when
  reproducibility matters;
- validation report URL or release asset path.

Add GitHub artifact attestations if the organization policy wants a stronger
supply-chain story. Attestations are additive to, not replacements for,
metadata sha checks.

## Publish, Rebuild, And Rollback

### Publish

1. A maintainer merges a formula or package-update PR.
2. `publish-bottles.yml` computes affected formulae and arches.
3. Each matrix entry builds a bottle and validates it.
4. Bottle bytes upload to GitHub Packages.
5. Formula bottle blocks and Kandelo sidecars are generated from the produced
   bytes.
6. The sidecar release `bottles-abi-v<N>` is updated under the publish
   concurrency lock.
7. A durable note or status names package count, pass/fail/skip counts, and the
   complete failure list.

### Rebuild

Use `rebuild-bottles.yml` for:

- new Kandelo ABI;
- formula build input changes;
- package cache-key bug fixes;
- dependency rebuilds;
- missing or corrupted sidecar metadata.

The workflow should skip matrix entries whose current metadata already matches
the computed cache key and bottle sha unless `force = true`.

### Rollback

Do not delete GitHub Packages bottle blobs as the normal rollback path. Prefer:

1. Revert the tap commit that selected the bad bottle block and sidecar.
2. Republish `Kandelo/metadata.json` with `status = "failed"` and last-green
   `fallback_*` fields for the bad entry.
3. If guest `brew install` has already selected the bad bottle through the
   formula, bump bottle `rebuild` or formula `revision` to move users forward.
4. Delete a package blob only for legal/security reasons, and record the
   deletion in the metadata report.

## Minimal Hello Milestone

The first implementation should target one formula:

```text
kandelo/core/hello
```

Acceptance:

- `Formula/hello.rb` builds a Wasm `hello` through the normal Kandelo SDK path.
- Trusted CI publishes a wasm32 ABI-15 bottle to GitHub Packages, or records an
  owner-approved GitHub Releases fallback.
- The formula has a generated `bottle do` block for `wasm32_kandelo`.
- `Kandelo/metadata.json`, `Kandelo/formula/hello.json`, and the link manifest
  validate.
- Host VFS planning rejects ABI mismatch before extraction.
- Host VFS building produces a Homebrew-prefix image without evaluating Formula
  Ruby.
- Node boots the image and `hello --version` runs through Kandelo `exec`.
- Browser boots the precomposed image if metadata marks it browser-compatible.
- A negative fixture with the wrong ABI fails with a message naming formula,
  bottle ABI, requested ABI, arch, and metadata source.

Suggested validation commands once implementation exists:

```sh
bash scripts/dev-shell.sh bash -c 'cargo test -p xtask homebrew'
bash scripts/dev-shell.sh bash -c 'cargo xtask homebrew-vfs-plan --metadata <metadata> --package hello --abi 15 --arch wasm32 --out target/hello.plan.json'
bash scripts/dev-shell.sh bash -c 'cargo xtask homebrew-vfs-build --plan target/hello.plan.json --out target/hello.vfs.zst --report target/hello.report.json'
node <harness> --vfs target/hello.vfs.zst /home/linuxbrew/.linuxbrew/bin/hello --version
./run.sh browser
```

## Node And Browser Validation

Node and browser are peers, but they do not need the same transport strategy in
the first milestone.

Node validation should cover:

- `brew --version` inside the Homebrew image when runtime bring-up is present;
- `brew install kandelo/core/hello` from the published bottle once guest
  install is implemented;
- host VFS planner and builder using sidecar metadata;
- `hello --version` through normal Kandelo process execution;
- ABI mismatch and missing bottle failures.

Browser validation should cover:

- launching a precomposed VFS image built from the same sidecar metadata;
- visible failure for missing VFS image, missing metadata, or ABI mismatch;
- no direct browser bottle download claim unless CORS/fetch behavior has been
  tested for the bottle host;
- no gallery entry unless every listed package has successful wasm32 metadata
  and `browser_compatible = true`.

For browser-facing behavior, `./run.sh browser` plus Playwright smoke coverage
is required before claiming browser support.

## Open Owner Decisions

1. Final tap owner/name: `Automattic/homebrew-kandelo`,
   `kandelo/homebrew-core`, or another organization.
2. Whether the first `hello` bottle may use a GitHub Releases `root_url`
   fallback if GitHub Packages upload support is blocked by Homebrew tag work.
3. Whether CI may push generated bottle metadata directly to tap `main`, or
   must open a bot PR.
4. Exact Homebrew patch shape for `wasm32_kandelo` and `wasm64_kandelo` tags.
5. Whether ABI should ever appear in Homebrew bottle tags or stay in sidecar
   metadata and release namespaces.
6. Whether GitHub artifact attestations are required before third-party taps
   are accepted by host VFS builders.
7. Which packages are allowed to claim `browser_compatible = true` in the first
   release.
8. Whether source-build formulae are deferred until after bottle-only install,
   or allowed in trusted CI for maintainers only.
