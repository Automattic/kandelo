## What changed (this bead)

The BEAM smoke already existed but its ring test called an **external `ring`
module** that is never compiled, so `ring:start()` died with
`{undef,[{ring,start,...}]}`. The ring test is replaced with a **self-contained
named-fun ring** driven through `-eval`, so all 5 tests exercise a real BEAM
program with no compilation step. `demo/ring.erl` and `demo/serve.ts` doc
comments are corrected to stop advertising the broken `ring:start()` path.

Root cause of "why not just compile ring.erl on Kandelo": the on-Kandelo Erlang
compiler crashes in `beam_asm` because `erlang:md5/1` returns **badarg on iolist
input** in this wasm BEAM build (it works only on a binary). Filed as **kd-qe2c**.

## maxAddr / memory (explicit — acceptance)

`serve.ts` sets an explicit guard, logged at launch:
`[setup] max_addr set to 0x3fdc0000 (page 16348) to protect thread region`.
Derivation: `MAX_PAGES(16384) − 2 − maxThreads(8)×pagesPerThread(4) − 2 = page
16348`; `maxAddr = 16348 × 65536 = 0x3fdc0000`, passed as `host.spawn(..., {
maxAddr })`. Reserves the top of the wasm32 address space for BEAM thread
channels/TLS so dirty-scheduler/async threads do not collide with the heap.

## Browser smoke — status: SKIPPED (documented boundary)

- Status: **skipped** — no browser BEAM harness exists. The erlang demo has only
  a Node launcher (`serve.ts` → NodeKernelHost); there is no BrowserKernelHost
  entry for BEAM.
- Reason/evidence: BEAM-in-browser needs a BrowserKernelHost path plus
  SharedArrayBuffer threads and the ~42 MB OTP install tree delivered to the
  browser VFS; that is materially heavier than this smoke and is not required by
  the bead's Node-smoke acceptance. Not forced (would otherwise mask the absence
  of a browser harness behind a stub).
- Candidate follow-up: a browser BEAM smoke once the OTP runtime data tree is a
  publishable sidecar (kd-1mr.6 runtime data-tree packages; kd-yuef sidecar
  publish). Recorded, not filed unilaterally.

## Reproducibility / limitation

The smoke PASSES when `erlang.wasm` + `erlang-install/` (the OTP tree) are present
locally, and SKIPS otherwise (pre-existing `hasErlang` contract; unchanged). The
abi release ships only `erlang.wasm`, not the OTP install tree, so CI/browser
without a staged OTP tree records the 5 tests as **skipped**, not failed. Making
the OTP tree fetchable is packaging work tracked by kd-1mr.6 / kd-yuef, not this
smoke bead. Artifacts used here were staged from the kd-xhdd worktree (same
convoy base, ABI 16): `erlang.wasm`, `erlang-install/`, `local-binaries/kernel.wasm`,
`sysroot/` (all gitignored build outputs; none committed).
